# 宠物消费管理系统 - 1人敏捷开发规范
本规范为单人开发场景定制，轻量化高效，核心要求为「开发 → 测试 → 推代码」，**测试是不可逾越的红线要求**

---
## 🎯 核心开发流程（必须严格执行）
```
需求确认 → 本地开发 → 【强制测试】 → 提交代码 → 推送远程 → 上线验证
```
**⚠️ 测试不通过绝对不能提交代码，这是第一准则**

---
## 📝 1. 开发环节规范
### 1.1 需求管理
- 所有需求/任务先在飞书任务/本地TODO记录，明确需求内容、验收标准，避免做无用功
- 需求变更必须同步更新记录，不做「脑袋里记的需求」
- 复杂功能先画简单流程图/接口定义，再写代码，避免走弯路

### 1.2 代码规范
- 后端遵循PEP8规范，接口遵循RESTful规范，统一返回格式
- 前端遵循Vue3风格指南，样式统一使用全局设计变量（`style.css`中的颜色、间距、圆角），禁止硬编码样式值
- 关键逻辑必须加注释，说明「做什么、为什么这么做」
- 依赖必须锁版本：后端提交`uv.lock`，前端提交`package-lock.json`，保证环境一致性

### 1.3 Git提交规范
- 提交信息必须带emoji前缀，清晰明了：
  | emoji | 用途 | 示例 |
  |-------|------|------|
  | ✨ | 新增功能/页面/接口 | ✨ 新增月度预算管理页面 |
  | 🐛 | 修复bug | 🐛 修复统计页金额计算错误 |
  | 💄 | UI/样式优化 | 💄 适配Notion风格全局样式 |
  | ♻️ | 代码重构/性能优化 | ♻️ 重构消费记录查询接口 |
  | 📝 | 文档更新 | 📝 更新开发规范文档 |
  | 🔧 | 配置/工具链调整 | 🔧 升级依赖版本 |

---
## ✅ 2. 强制测试环节（必须全部满足才能提交）
### 2.1 后端测试标准
1. 代码写完本地启动服务，所有新接口/修改过的接口手动调用验证，参数正确和错误情况都要测
2. 数据库迁移脚本本地跑通，测试增删改查操作正常
3. 单元测试用例跑通，核心逻辑覆盖率100%

### 2.2 前端测试标准
1. 必须跑`npm run build`编译通过，没有任何报错
2. 所有新增/修改的功能点手动走一遍，正常流程和边界情况都要覆盖
3. 页面在1440px和1920px宽度下显示正常，样式无错乱
4. 响应式适配正常（移动端无明显错位）

### 2.3 全流程冒烟测试
每次提交前必须跑通核心主流程：
> 创建客户 → 创建宠物 → 创建消费记录 → 统计页数据展示正确 → 预算页计算正确
确保主流程全链路通畅，不能出现「改了A功能把B功能搞坏」的情况

---
## 🚀 3. 提交推送规范
1. 提交前先拉取最新的`dev`分支代码，解决所有冲突再提交
2. 检查`git diff`，确认没有提交敏感信息（密码、密钥、本地配置、测试数据）和无用文件（node_modules、本地缓存文件等）
3. 本地分支：只用`dev`分支开发，`main`分支作为稳定版本分支，只有上线后才合并，禁止直接提交到`main`
4. 推送前再跑一遍编译/测试，确认没问题再推远程

---
## 🔴 4. 上线规范
1. 上线前必须备份数据库，特别是有表结构变更的时候
2. 部署后第一时间做健康检查：
   - 后端服务正常启动，接口可访问
   - 前端页面正常打开，没有资源加载错误
   - 核心流程走一遍，确保功能可用
3. 上线后观察10分钟，查看日志有没有报错

---
## ❌ 5. 红线禁令（绝对不能碰）
1. ❌ 禁止代码不编译、不测试就提交推送，这是最高优先级要求
2. ❌ 禁止提交任何敏感信息（密码、密钥、内部配置）到代码仓库
3. ❌ 禁止不备份直接操作生产数据库、直接修改线上代码
4. ❌ 禁止没确认需求就开发，避免做无用功
